# Patent Specification (Draft) — Automated Derivation and Compilation of Multi-Tenant Regulatory Compliance Obligations into Machine-Enforceable Agent Policy

Status: draft, narrowed claim. Supersedes any prior broader draft of this document.

## 0. Scope of the Claim (read this first)

This filing claims **one thing**: an automated method and system for deriving
machine-enforceable policy (Rego) from regulatory compliance obligations held
in a multi-tenant GRC (governance, risk, and compliance) system, with
per-obligation provenance, and packaging the result for consumption by an
independent, pre-existing agent policy enforcement runtime.

It does **not** claim, and must not be read as claiming:

- Pre-execution interception of AI agent tool/action calls.
- Evaluation of Rego/OPA, YAML, or Cedar policy at runtime.
- Ed25519-based cryptographic identity or signing of decisions.
- Hash-chained, tamper-evident audit/decision records ("decision bill of
  materials").
- Compilation of an already-structured rule specification into Rego syntax
  (e.g. mapping a rule-plan object model to Rego source text).

All of the above are provided today, under the MIT license, by the
**Microsoft Agent Governance Toolkit** (github.com/microsoft/agent-governance-toolkit),
and — as detailed in §1 — that toolkit already includes its own structured
rule-to-Rego compiler. CompliVibe is a policy *author*, not a policy
*enforcer* or a policy *compiler of already-structured rules*. The invention
claimed here sits strictly upstream of both: turning unstructured/semi-structured
regulatory text and metadata into the structured constraint specification that
any Rego compiler (the toolkit's own, or this system's) can then render.

## 1. Prior Art Disclosure

### 1.1 Microsoft Agent Governance Toolkit (MIT License)

Publicly available at github.com/microsoft/agent-governance-toolkit. Verified
directly against the toolkit's source (not assumed from documentation) as of
this filing. It provides, and this patent does not claim:

1. **Pre-execution action interception.** A manifest-driven intervention-point
   model (`pre_tool_call`, `post_tool_call`, and others) resolved from a
   `governance.yaml` chain by the toolkit's manifest-resolution layer
   (`agt.manifest_resolution`), which builds a canonical policy input object
   per the toolkit's own `AGT-SNAPSHOT` specification before any policy is
   consulted.

2. **OPA/Rego/Cedar policy evaluation.** A dual-mode Rego evaluator
   (`agentmesh.governance.opa.OPAEvaluator`) supporting a remote OPA server
   (`POST /v1/data/...`) or a local `opa eval` subprocess, wrapped by an
   `OPAPolicyBackend` adapter that satisfies the toolkit's own
   `ExternalPolicyBackend` structural protocol (`name`, `evaluate(action,
   context) -> PolicyDecisionResult`, `healthy()`) and registers with the
   toolkit's `BackendRegistry`. The toolkit additionally exposes a
   framework-agnostic ASGI `PolicyProviderHandler` (`/check`, `/health`,
   `/policies`) so API gateways can call into it directly. Cedar policy
   evaluation is provided separately (e.g. via `cedarling-agentmesh`).

3. **A structured-rule-to-Rego compiler.** The toolkit's own
   `policy-engine/generator/acs_generator/rego_builder.py` already compiles an
   object model of rules (`PolicyPlan` / `RulePlan`, each with an intervention
   point, a decision, conditions, and optional transform effects) into valid
   Rego source, including deterministic decision-severity ordering
   (`deny > escalate > transform > warn > allow`) and per-package emission
   (`package agent_control_specification.<slug>`). **This is prior art that
   materially narrows the present claim**: automated compilation of an
   already-structured rule specification into Rego syntax is not novel. The
   novelty asserted in §3 below is confined to the step that produces that
   structured specification in the first place, from unstructured regulatory
   source material, with cross-tenant provenance — a step the toolkit's
   generator does not perform and does not attempt.

4. **Ed25519 cryptographic identity and offline-verifiable receipts.** The
   toolkit's `mcp_receipt_governed` package (`McpReceiptAdapter`, plus
   module-level `sign_receipt`/`verify_receipt`/`verify_receipt_chain`
   functions) signs per-tool-call receipts with an Ed25519 key over a
   canonical JSON payload, supplied by the caller as `signing_key_hex` — the
   key is never fetched or generated by the toolkit's own infrastructure —
   and chains receipts via a `parent_receipt_hash` field for tamper
   detection. **This is not merely documented but verified installable and
   working**: this repository installs the package directly and calls it
   for real (see `core-side-patch/services/receipts.py`, ASSUMPTIONS.md);
   a live test signed a chain of receipts, tampered with one after signing,
   and confirmed `verify_receipt_chain` correctly flagged both the invalid
   signature and the resulting broken hash link at the next receipt. The
   customer-side key-custody design in Claim 4 below describes this real,
   exercised behavior, not an assumed one.

5. **A tamper-evident audit chain ("decision bill of materials").** The
   combination of (2) and (4) above already gives any adopter a verifiable,
   chained record of what was decided and why, independent of this filing.

None of (1)–(5) is reproduced, reinvented, or claimed by this patent. Where
this repository's code touches any of these concerns, it does so exclusively
by calling into the toolkit's published extension points (the
`ExternalPolicyBackend` protocol and `McpReceiptAdapter`), never by
reimplementing their behavior.

### 1.2 Manual regulatory-to-Rego authoring (status quo)

Absent this invention, translating a regulatory obligation (e.g. "core banking
data must not leave the jurisdiction of collection" under an RBI data-localization
circular) into enforceable Rego is performed by a human policy engineer, once
per tenant, per obligation, per amendment. See Workstream M
(`tests/benchmark/PATENT_TECHNICAL_EFFECT.md`) for a reproducible comparison.

## 2. Technical Problem

Regulatory obligations live in a GRC system (CompliVibe's obligation graph) as
unstructured or semi-structured records: free-text obligation language,
jurisdiction and framework metadata, and linked controls — authored and
maintained by compliance and legal personnel, not software engineers. Policy
enforcement runtimes (including the Agent Governance Toolkit) require a
formally specified, machine-executable policy artifact (Rego). There is a
representational gap between these two forms, and today it is closed only by
manual translation:

- A compliance analyst identifies which obligations apply to a given AI agent
  system and tenant.
- A policy engineer reads each obligation's natural-language text and
  hand-derives a constraint (a financial limit, a geographic restriction, a
  data-scope boundary, an approval requirement).
- The engineer hand-writes Rego expressing that constraint, once per tenant,
  and must remember to revisit it whenever the source obligation changes.

This process does not scale across CompliVibe's multi-tenant customer base,
is error-prone (a missed clause, an incorrectly scoped jurisdiction, a stale
Rego file after the obligation is amended upstream), and creates silent drift
between what compliance has recorded as a legal obligation and what is
actually enforced at runtime.

## 3. Technical Solution

An automated derivation-and-compilation pipeline (Workstream B,
`core-side-patch/services/derivation_engine.py`) that:

1. Takes one or more structured compliance obligation records (from
   CompliVibe's obligation graph, or a guardrail definition input referencing
   such obligations) as input.
2. Derives a normalized **constraint specification** from the obligation
   text and metadata: financial limits, geographic scope, data scope, and
   approval requirements, each retaining a link back to the specific source
   obligation(s) it was derived from (provenance).
3. Compiles the constraint specification into valid Rego policy text, scoped
   to a per-tenant package namespace (`complivibe.guardrails.org_{org_id}`),
   ready to be handed to the enforcement runtime's own evaluation path (the
   toolkit's `ExternalPolicyBackend` / OPA evaluator described in §1.1(2)) via
   this system's `CompliVibePolicyProvider` (Workstream D). **This step,
   taken alone, is not asserted as novel** — see §1.1(3): the toolkit already
   has its own compiler from a structured rule model to Rego. It is included
   here only because a constraint specification with no Rego rendering at
   all would not be "machine-enforceable policy," not because rendering it
   is itself the inventive contribution. The inventive step is exclusively
   step 2 above.
4. Persists the guardrail record (`ai_policy_guardrails`) with an explicit,
   queryable link from compiled Rego back to the obligation(s) that produced
   it, so that when an obligation is amended upstream, affected guardrails
   can be identified and recompiled (`POST /ai-guardrails/{id}/compile-rego`)
   without a human re-reading the regulation from scratch.

The system does not perform interception, runtime evaluation, signing, or
audit-chaining itself — those are delegated to the toolkit per §1.1, with
CompliVibe's core only ever receiving and storing already-signed receipts (see
ASSUMPTIONS.md for the exact key-custody boundary).

## 4. Technical Effect

Benchmarked and documented reproducibly in
`tests/benchmark/PATENT_TECHNICAL_EFFECT.md` (Workstream M): given a
representative regulatory obligation, the manual-authoring path is compared
step-by-step against the automated path, with the automated path's output
validated for syntactic correctness and expected allow/deny semantics
against a local Rego evaluator (test-only `opa eval`, not a live
deployment). The claimed technical effect is attributable specifically to
automating the *derivation* step (obligation text → structured, provenance-
tagged constraint specification) — the step from a structured specification
to rendered Rego text was already automatable using existing tools (see
§1.1(3)) and is not where the measured reduction in manual, error-prone
steps originates. The effect is a measurable reduction in the number of
manual steps and error-prone judgment calls required to go from "obligation
recorded in the GRC system" to "obligation enforced," not merely an
assertion of convenience.

## 5. Claims (sketch)

**Claim 1 (method).** A computer-implemented method for compiling a
regulatory compliance obligation, stored in a multi-tenant governance, risk,
and compliance data store, into machine-enforceable policy, the method
comprising: retrieving one or more structured obligation records associated
with a tenant and an AI agent system; deriving therefrom a constraint
specification comprising at least one of a financial limit, a geographic
scope, a data scope, or an approval requirement, each element of the
constraint specification retaining a reference to its source obligation
record; compiling the constraint specification into policy text in a
declarative policy language, scoped to a tenant-specific policy namespace;
and persisting the compiled policy text together with the retained references
to its source obligation record(s), such that a subsequent amendment to a
source obligation record can be programmatically traced to every compiled
policy artifact derived from it.

**Claim 2 (system).** A system comprising the method of Claim 1, further
comprising an interface conforming to a third-party agent-action policy
enforcement runtime's published external-policy-backend extension point,
whereby the compiled policy text is supplied to that runtime for evaluation
without the system performing action interception, policy evaluation, or
cryptographic receipt signing itself.

**Claim 3 (dependent).** The method of Claim 1, wherein the compiled policy
text is Rego, and the tenant-specific policy namespace is a distinct Rego
package per tenant, such that policy compiled for one tenant cannot be
evaluated against an action originating from another tenant.

**Claim 4 (dependent).** The method of Claim 1, further comprising receiving,
from the policy enforcement runtime, a cryptographically signed decision
receipt produced using a private signing key held exclusively within the
runtime's own deployment environment, and storing that receipt without ever
receiving or storing the private signing key.

## 6. Alignment with Patentability Requirements (Section 3(k) and analogues)

The claimed invention is not merely an abstract algorithm or a computer
program "as such." It solves a concrete technical problem — the
representational mismatch between unstructured regulatory text and a
declarative policy execution language — with a technical effect that is
externally measurable (§4): a working, syntactically valid, semantically
correct machine-enforceable artifact produced without per-tenant manual
authoring. The claimed system produces a change in what a computer system
does (a runtime now enforces a rule that previously existed only as
unexecuted text), which is the kind of technical contribution recognized as
patent-eligible subject matter notwithstanding the involvement of software,
consistent with the "technical effect"/"technical contribution" framing used
in the corresponding P2 filing for this portfolio. The invention is narrowly
scoped specifically to avoid claiming the enforcement, signing, and
audit-chain functionality already disclosed as prior art in §1.1, so as not
to claim what is already public, MIT-licensed, and non-novel.

## 7. Branding and Disclosure Boundary (internal note, retained for filing integrity)

The Microsoft Agent Governance Toolkit is named explicitly in this document,
as legally required for prior-art disclosure. It is intentionally **not**
named in source code, log output, API responses, or any customer-facing
surface anywhere else in this repository — those refer to it only generically
("the policy enforcement runtime") or via its literal Python import
(`agent_governance_toolkit` / the toolkit's actual package names such as
`agentmesh`, `mcp_receipt_governed`). This is a deliberate, legitimate split
between a legal disclosure obligation (this document) and a product-branding
decision (everywhere else), not an inconsistency.
